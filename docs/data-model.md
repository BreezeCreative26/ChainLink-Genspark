# ChainLink ‚Äî Data Model

This document describes the MVP database schema implemented in
`supabase/migrations/`. It is the detailed companion to the "Data Model
Overview" section of `docs/ARCHITECTURE.md` ‚Äî read that first for the
reasoning, this file for the specifics.

## Conventions

- Primary keys are `uuid`, generated with `gen_random_uuid()`.
- Enums are implemented as `text` + `check` constraints, not native Postgres
  enum types ‚Äî easier to extend later without a schema migration that
  touches every dependent column.
- Two cross-cutting columns appear on every content table (`milestones`,
  `tasks`, `notes`, `documents`, `activity_logs`):
  - **`visibility`**: `'shared' | 'internal'` ‚Äî the two-tier model from
    `docs/OPERATING_MODEL.md`. There is no third "guest" tier; guests see
    `shared` content on their one chain like anyone else.
  - **`source`**: `'manual' | 'proxy' | 'system'`, plus (where relevant)
    `recorded_by_participant_id` / `created_by_participant_id` and
    `on_behalf_of_participant_id` ‚Äî attribution for proxy-mode updates.

## Entities

### Business-workspace layer

- **`organisations`** ‚Äî a firm (estate agency or conveyancing firm).
  Carries billing/subscription fields (`plan`, `subscription_status`,
  `stripe_customer_id`, `stripe_subscription_id`, `current_period_end`,
  `trial_ends_at`) structured to match what a future Stripe integration
  would populate ‚Äî see `docs/DECISIONS.md` ("Commercial / billing
  scaffolding"). Plan *definitions* (limits, features, pricing) live in
  `src/config/plans.ts`, not the database.
- **`branches`** ‚Äî optional sub-tenancy within an organisation. Nullable
  FK from `memberships`, so a firm can stay flat until it needs branches.
- **`profiles`** ‚Äî extends `auth.users`. One identity type for everyone;
  guest vs. professional is never encoded here (see `ARCHITECTURE.md`,
  "Auth Model").
- **`memberships`** ‚Äî profile ‚Üî organisation (+ optional branch), with a
  business-workspace role (`owner` / `admin` / `agent` / `conveyancer` /
  `staff`). This is distinct from a person's role *on a chain*.

### Chain-workspace layer

- **`chains`** ‚Äî the canonical workspace. Carries `chain_ref`, the
  human-readable, unique Chain ID (e.g. `CHAIN-7K2F9X`), generated by
  `generate_chain_ref()` with a collision-retry loop.
- **`properties`** ‚Äî scoped to a single chain (not shared across chains,
  even for the same physical address, to keep chain data self-contained).
- **`chain_nodes`** ‚Äî one row per property transaction (a seller‚Üíbuyer
  sale) within a chain. See "Chain topology" below.
- **`chain_participants`** ‚Äî a profile's role and access mode on a specific
  chain. This is where `access_mode` (`proxy` / `guest` / `connected`) and,
  for connected participants, `organisation_id` live ‚Äî this is the single
  most important table in the schema (see "Access model" below).
- **`invitations`** ‚Äî pre-acceptance state for a pending participant.
  Becomes a `chain_participants` row only on acceptance. Status lifecycle:
  `invited` ‚Üí `viewed` ‚Üí `accepted` (guest/proxy-side outcome) or `linked`
  (matched into an existing firm workspace); `declined` and `inactive`
  (revoked/expired) are also terminal. See `docs/DECISIONS.md`
  ("Invitation system") for why this six-state lifecycle lives here and
  not on `chain_participants`.

### Content tables

- **`milestone_templates`** ‚Äî reusable milestone sets, either global
  (`organisation_id is null`) or firm-specific. Carries `guest_confirmable`,
  inherited by milestones created from the template.
- **`milestones`** ‚Äî chain-wide or per-node (`chain_node_id` nullable).
  `guest_confirmable` (boolean) marks milestones a guest is allowed to mark
  complete themselves (e.g. "ID documents submitted") ‚Äî see "Guest
  confirmation safety" below.
- **`tasks`** ‚Äî assignable to a `chain_participants` row; defaults to
  `internal` visibility (most tasks are a firm's own working items).
- **`notes`** ‚Äî free text, defaults to `internal`. Also doubles as guest
  "comments" when `visibility = 'shared'` ‚Äî see `docs/DECISIONS.md`
  ("Guest experience") for why this isn't a separate table.
- **`documents`** ‚Äî metadata only; binary content lives in Supabase
  Storage at `storage_path` (bucket `chain-documents`, 20MB limit, fixed
  MIME allowlist), with object-level policies mirroring this table's
  visibility rules exactly (`0013_document_storage.sql`). `category` is a
  fixed 8-value enum (memorandum of sale / ID docs / sales forms / EPC /
  contract pack / search results / mortgage offer / other,
  `0015_document_categories_and_rules.sql`). Upload is further restricted
  by role: guests may only use `id_docs`/`sales_forms`/`other`;
  `contract_pack`/`search_results` require a conveyancer role ‚Äî see
  `docs/DECISIONS.md` ("Document handling") for the explicit legal-review
  flag on this mapping, and on the fact that guest uploads are necessarily
  `shared` rather than per-participant private.
- **`activity_logs`** ‚Äî generic, immutable audit trail (`entity_type` +
  `entity_id` rather than one row-type per action). Powers both the casual
  shared-only "Activity" feed every participant sees and the fuller,
  professional-only audit log at `/chains/[id]/audit` (same table, no
  separate audit schema ‚Äî the audit view is a different query/permission
  gate over the same rows, not different data).
- **`notifications`** ‚Äî per-profile, not per-participant, since a person
  should be notified as themselves regardless of which chain triggered it.

## Chain Topology (resolves the open question in `docs/DECISIONS.md`)

Chains branch in the real world ‚Äî most commonly, a seller is simultaneously
buying their own onward property. `chain_nodes` models this as a **tree**:
each node may declare a single `depends_on_node_id`, meaning "this
transaction cannot complete until that one does."

This was chosen over two alternatives:

- **Strict linear list** ‚Äî rejected, since it can't represent the common
  onward-purchase case at all.
- **Full multi-parent DAG** (a separate edges table, arbitrary dependency
  graphs) ‚Äî rejected for MVP as more structure than any current product
  requirement needs. If a real scenario later demands multiple dependencies
  per node, `depends_on_node_id` can be promoted into a `chain_node_links`
  edges table without breaking anything referencing the current column.

## Access Model, In Schema Terms

Three access modes, all on `chain_participants`:

- **`proxy`** ‚Äî requires `proxy_manager_participant_id` (enforced by a check
  constraint). Updates made on this participant's behalf are recorded with
  `source = 'proxy'` and `on_behalf_of_participant_id` set.
- **`guest`** ‚Äî no organisation, no proxy manager. Scoped to exactly one
  chain via the normal `chain_participants` row ‚Äî nothing additional is
  needed, since RLS scopes all access through this table already.
- **`connected`** ‚Äî requires `organisation_id` (enforced by a check
  constraint). This is what makes a chain appear in a firm's business
  workspace: the dashboard query is, in essence, "chains where I have a
  `connected` participant row."

**Multiple firms per chain** (an open question in `docs/DECISIONS.md`,
now resolved): because `organisation_id` lives on the *participant*, not
the chain, two firms can independently be `connected` to the same chain.
Internal-visibility rows (`tasks`, `notes`, etc.) each carry their own
`organisation_id`, so RLS scopes each firm's internal content to itself
automatically ‚Äî this is exercised directly in `supabase/seed.sql`.

**Guest-to-connected transition** (also resolved): since guests and
professionals share one `profiles` identity, no separate "claim" table is
needed. When a guest professional later joins a firm, their existing
`chain_participants` rows are simply queried by `profile_id` and offered
for linking (updating `access_mode` to `connected` and setting
`organisation_id`).

## Guest Confirmation Safety

Guests need to be able to confirm certain things (e.g. "I've submitted my
ID documents") without gaining general edit rights over milestones ‚Äî a
buyer marking "Exchange of contracts" complete themselves would be a real
problem. Two layers enforce this:

1. **RLS** (`milestones_update`, `0012_guest_capabilities.sql`) ‚Äî a
   baseline "any chain member can update a shared milestone" rule.
2. **A `BEFORE UPDATE` trigger** (`enforce_guest_milestone_confirmation`)
   that, specifically when the acting user's access_mode on that chain is
   `guest`, additionally requires: the row must already be
   `guest_confirmable` and `shared`; the new status must be `completed`;
   and every other column (title, due date, visibility, template,
   organisation, etc.) must be unchanged from the existing row.

Non-guests are unaffected by the trigger ‚Äî it returns immediately for
them. This is deliberate defense-in-depth: the application's service layer
(`src/server/services/milestones.ts`) only ever sends a narrow update, but
the trigger holds even if that guarantee were ever bypassed (a bug, a
future code path, a direct API call).

## Row Level Security

RLS is enabled on every table (`supabase/migrations/0008_row_level_security.sql`).
Three helper functions back the policies:

- `is_chain_member(chain_id)` ‚Äî true if the caller has an active
  `chain_participants` row on that chain.
- `is_org_member(organisation_id)` ‚Äî true if the caller has an active
  `memberships` row in that organisation.
- `shares_chain_with(profile_id)` ‚Äî true if the caller shares any active
  chain with the given profile (used to let chain participants see each
  other's names without exposing the whole `profiles` table).

Content tables follow one rule: **`shared` rows are visible to any chain
member; `internal` rows are visible only to members of the owning
organisation.** This is the database-level enforcement of
`docs/OPERATING_MODEL.md`'s "Internal vs Shared Visibility" section.

This is an MVP baseline, not the final security posture ‚Äî see
`docs/ROADMAP.md` Phase 3 ("Reporting + Hardening") for the planned review
pass (finer-grained role checks, e.g. restricting who can mark a legal
milestone complete).

## Seed Data

`supabase/seed.sql` seeds one chain that exercises all three access modes
and both firms-on-one-chain and internal/shared visibility simultaneously:

- an agent (connected, Blake & Co.)
- a seller (proxy-managed by that agent)
- a buyer (guest, no business account)
- a buyer's conveyancer (connected, a second firm ‚Äî Fenwick Conveyancing)
- two linked `chain_nodes` (the sale, and the seller's dependent onward
  purchase)

Run it against a local Supabase instance with `supabase db reset` (which
runs all migrations, then `seed.sql`).

## Full-chain progression

Migration `0018_full_chain_progress.sql` completes the open-chain progression model. Every `chain_node` receives the canonical UK conveyancing stages, while `seller_participant_id` and `buyer_participant_id` identify the private parties on that transaction. Accepted buyer/seller invitations attach automatically only when one vacant side is unambiguous; managers assign parties explicitly for branching chains. The chain creator is the initial administrator even when their participant access mode is `guest`, provided their participant record remains active. RLS and validation triggers continue to enforce chain membership, side roles, and shared/internal visibility.
